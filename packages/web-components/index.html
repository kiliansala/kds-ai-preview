Getting Started

Instala, configura y empieza a usar los componentes de KDS en tu proyecto.

Overview

KDS proporciona web components construidos con LIT 3.x, respaldados por design tokens extraidos de Figma. Cada componente se valida automaticamente contra su contrato de diseno para garantizar 100% fidelidad.

  • Figma as SSOT — Tokens y contratos extraidos via MCP
  • Framework Agnostic — Web components nativos con wrappers para React, Angular y Blazor
  • 100% Design Fidelity — Validacion automatica diseno ↔ codigo
  • DTCG Tokens — 90+ tokens en formato estandar
  • TypeScript First — Tipado completo e IntelliSense

Packages

El monorepo contiene 5 packages:

Package Descripcion
@kds/tokens Design tokens en formato DTCG (tokens.json, tokens.css, TypeScript)
@kds/web-components Web components LIT validados contra Figma
@kds/react React wrappers via @lit/react
@kds/angular Angular 20/21 standalone components

Accessibility

All KDS components follow WCAG 2.1 AA standards to ensure an inclusive experience for all users.

WCAG 2.1 AA

Our Commitment

Every component in the KDS library is designed and validated to meet WCAG 2.1 Level AA accessibility standards. We believe that accessibility is not optional—it's a fundamental requirement for all web experiences.

Four Principles (POUR)

Our components follow the four core principles of accessibility:

Perceivable

Information and UI components must be presentable to users in ways they can perceive.

Operable

UI components and navigation must be operable by all users.

Understandable

Information and operation of UI must be understandable to all users.

Robust

Content must work with a wide variety of user agents and assistive technologies.

Automated Validation

Every component passes through automated accessibility validation before release:

# Run accessibility validation
npm run validate:a11y --workspace=@kds/web-components

# Run all validations (design contract + a11y)
npm run validate:all --workspace=@kds/web-components

What we check automatically:

  • ✅ ARIA attributes and roles
  • ✅ Keyboard navigation support (Tab, Enter, Space)
  • ✅ Focus indicators (:focus-visible)
  • ✅ Touch target sizes (minimum 44x44px)
  • ✅ Disabled state handling
  • ✅ Screen reader compatibility
  • ⚠️ Color contrast ratios (manual verification required)

Keyboard Navigation

All interactive components are fully keyboard accessible:

Key Action
Tab Move focus to next interactive element
Shift + Tab Move focus to previous interactive element
Enter Activate button or link
Space Activate button
Esc Close modals, dropdowns, and overlays

Screen Reader Support

All components work seamlessly with popular screen readers:

  • VoiceOver (macOS, iOS) - Full support
  • NVDA (Windows) - Full support
  • JAWS (Windows) - Full support
  • TalkBack (Android) - Full support

We use semantic HTML and proper ARIA attributes to ensure all components are properly announced.

Best Practices

When using KDS components, follow these accessibility best practices:

  1. Provide meaningful labels - Always use clear, descriptive text for buttons and form elements
  2. Use aria-label for icon-only buttons - Icon buttons without text must have an accessible name
  3. Maintain logical focus order - Ensure tab order follows the visual flow of your page
  4. Test with keyboard only - Disconnect your mouse and navigate using only the keyboard
  5. Test with screen readers - Verify all content is properly announced
  6. Provide alternative text - All images and icons should have descriptive alt text
  7. Don't rely on color alone - Use icons, text, or patterns in addition to color

Full keyboard support is built-in:

Key Action

Focus Indicator: The button displays a visible focus ring using :focus-visible, which only appears during keyboard navigation (not when clicked with a mouse).

Screen Reader Support

The button is fully compatible with all major screen readers:

  • Semantic HTML: Uses native <button> element for proper announcement
  • Role: Implicit role="button" from native element
  • State: Disabled state is properly announced via aria-disabled
  • Labels: Text content or aria-label provides accessible name

Icon-Only Buttons

When using icon-position="only", an aria-label is required:

<!-- ❌ Bad: No accessible name -->
<kds-button icon-position="only">
  <svg slot="icon">...</svg>
</kds-button>

<!-- ✅ Good: Has aria-label -->
<kds-button icon-position="only" aria-label="Delete item">
  <svg slot="icon">...</svg>
</kds-button>

Disabled State

Disabled buttons are properly handled:

  • Not focusable: Cannot receive keyboard focus
  • No interaction: Click events are prevented
  • Announced: Screen readers announce "disabled" or "dimmed"
  • Visual feedback: Reduced opacity (60%) and no pointer cursor

Best Practices

✅ Do

  • Use clear, descriptive button text that explains the action
  • Provide aria-label for icon-only buttons
  • Maintain logical tab order in your page
  • Use appropriate hierarchy for visual importance
  • Test with keyboard navigation (disconnect mouse)
  • Test with a screen reader (VoiceOver, NVDA)

❌ Don't

  • Don't use icon-only buttons without aria-label
  • Don't use vague text like "Click here" or "Learn more" without context
  • Don't rely on color alone to convey meaning (use icons + text)
  • Don't remove focus indicators (they're required for accessibility)
  • Don't use small touch targets on mobile (prefer lg or xl)

WCAG 2.1 AA Compliance

The Input Field component meets all WCAG 2.1 Level AA accessibility requirements:

✅ Semantic HTML ✅ Screen Reader Compatible ✅ WCAG 2.1 AA

ARIA Attributes

The input field uses proper ARIA semantics to communicate state and structure:

  • Label: Native <label> element provides an accessible name; falls back to aria-label when no label prop is set
  • Error state: aria-invalid="true" on the native <input> when destructive is active
  • Error message: aria-describedby links the input to the hint text element, which serves as the error message
  • Help button: Native <button> with aria-label="Help" and type="button"
  • Decorative icons: Leading icon, alert icon, and help icon SVGs have aria-hidden="true"
<!-- ❌ Bad: error only conveyed via color, no accessible message -->
<kds-input-field destructive></kds-input-field>

<!-- ✅ Good: aria-invalid + error message via hint -->
<kds-input-field
  label="Email"
  hint="This email is already in use."
  destructive>
</kds-input-field>

<!-- ✅ Good: aria-label when no visible label -->
<kds-input-field
  placeholder="Search…"
  aria-label="Search">
</kds-input-field>

Keyboard Navigation

The component uses native elements, inheriting full browser keyboard support:

Key Action
Tab Move focus to the input field
Tab (again) Move focus to the help button (if visible)
Shift + Tab Move focus backwards
Enter or Space Activate help button (fires kds-help-click)
Any character key Type into the input (fires kds-input)

Focus Indicator: A 4px ring appears on the input wrapper via :focus-within and the inner input uses :focus-visible. Brand purple (#F4EBFF) in normal state; error red (#FEF3F2) in destructive state. Both meet the WCAG 2.4.7 Focus Visible requirement.

Screen Reader Support

The input field is compatible with all major screen readers (NVDA, JAWS, VoiceOver, TalkBack):

  • Label: Announced as the accessible name of the input via native <label> association
  • Placeholder: Used as aria-label fallback when no label prop is provided
  • Error state: aria-invalid="true" causes screen readers to announce the field as invalid; aria-describedby reads the error message automatically
  • Disabled state: Native disabled attribute causes screen readers to announce the field as dimmed/unavailable
  • Help button: Announced as "Help, button" via native <button> + aria-label
  • Mobile input type: input-type prop maps to the native type attribute, triggering the correct mobile keyboard (email, numeric, telephone, etc.)

Disabled State

When disabled is set, the following behaviors apply:

  • Background changes to #F9FAFB (Gray/50) to signal non-interactivity
  • Label, hint, and icon colors shift to reduced-contrast gray
  • Screen readers announce the field as "dimmed" or "unavailable"
  • The field is removed from the tab order

Best Practices

Do

  • Always provide a visible label so screen readers announce the field's purpose
  • Use hint to provide error messages alongside destructive — this connects the message via aria-describedby
  • Use input-type to specify the correct HTML input type (email, tel, number) for mobile keyboard optimization
  • Use aria-label only when no visible label can be provided (e.g., search bar)

Don't

  • Don't rely on placeholder text alone to communicate the field's purpose — placeholders disappear on input and are not reliably announced by screen readers
  • Don't use destructive without a hint message — error state conveyed only via color fails WCAG 1.4.1 (Use of Color)
  • Don't disable inputs without explaining why in nearby text — users need to understand the condition
  • Don't wrap kds-input-field inside a <label> — the component manages its own internal label association
